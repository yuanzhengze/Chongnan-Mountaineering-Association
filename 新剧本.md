# **ARG 剧本：冰封的坐标 (Project: Frozen Coordinates) v2.0**

## **1\. 核心逻辑重构**

**玩家任务**：通过网络考古手段，利用“两点一线一圆”的几何逻辑，计算出 2014 年登山队遇难的精确坐标。

解谜公式：

$$最终坐标 \= (起点 P1 \+ 方向向量 \\vec{P1P2}) \\cap (距离限制 R)$$

* **点 P1 (起点)**：登山队最后一次发送信号的 GPS 位置（藏在 Base64 日志中）。  
* **点 P2 (参照点)**：地标“鹰嘴岩”的位置（藏在相册源码的注释中）。  
* **半径 R (距离)**：日志中提到的移动距离（200m）。

**2\. 页面流程与线索详解**

### **A. 入口：错误的重定向**

* **操作**：玩家在 university.html 点击“登山社”。  
* **表现**：出现 404 错误页，随后 URL 自动跳转至 index.html（2014年旧版）。  
* **氛围**：旧版网页顶部有一条滚动的 JS 跑马灯：“服务器维护通知：本站将于 2014/07/16 停止动态更新。”

### **B. 线索一：隐藏在 Source Code 中的“鹰嘴岩” (P2)**

* **页面**：gallery.html (活动相册)  
* **表象**：页面看似正常，展示了几张风景照。有一张照片加载失败（显示破损图标），标题是“未名峰北坡景观”。  
* **动作**：玩家习惯性按 F12 或右键“查看网页源代码”。  
* **发现**：在那个破损图片的 \<img\> 标签旁，有一段被开发者留下的 HTML 注释：  
* **获得信息**：**点 P2 (鹰嘴岩)** 的精确经纬度。

### **C. 线索二：URL Hacking 与未发出的博文**

* **页面**：log\_view.html (内部日志)  
* **表象**：页面显示了 ID 为 101, 102, 103 的几篇日志。URL 栏显示为 .../log\_view.html?id=103。  
* **动作**：  
  * 玩家发现 103 号日志发布于 7月14日，内容一切正常。  
  * 玩家尝试手动修改浏览器地址栏的参数，将 id=103 改为 id=104。  
* **发现**：  
  * 页面加载出一个没有 CSS 样式的“草稿预览”模式。  
  * 标题：“Connection Error / 自动保存草稿”。  
  * 正文部分不是汉字，而是一大串乱码字符。  
* **获得信息**：通往真相的密文。

### **D. 线索三：Base64 解码与“起点” (P1)**

* **谜题**：玩家复制上述乱码，识别出这是 **Base64** 编码（特征：字母数字混合，结尾可能有 \= 号）。  
* **解码内容**：  
  Plaintext  
  SYSTEM\_LOG\_DUMP:  
  时间: 2014-07-15 14:45  
  电量: 4%  
  状态: 极度低温，触屏失灵  
  \-------------------  
  GPS\_LAST\_PING: \[34.2140N, 108.5500E\] (Accuracy: High)  
  文本记录:  
  风太大了，我们在 P1 点（GPS已记录）无法扎营。  
  队长决定，全员向 P2 (鹰嘴岩) 方向移动。  
  我们只走了 200米 就不得不停下... 前面有... (数据截断)

* **获得信息**：  
  1. **点 P1 (最后记录点)**：34.2140N, 108.5500E。  
  2. **移动意图**：从 P1 往 P2 (鹰嘴岩) 走。  
  3. **距离 R**：走了 200米。

**3\. 终局：三角定位与真相**

玩家现在拥有了所有数据，需要进行一次“纸上谈兵”的硬核推理。

### **逻辑推导过程：**

1. **确定方向**：根据 P1 (34.2140, 108.5500) 和 P2 (34.2158, 108.5521) 两个坐标，可以画出一条直线向量。这意味着队伍是沿着这条线走的。  
2. **确定距离**：日志说“走了 200米”。  
3. **确定落点**：以 P1 为圆心，200米为半径画圆。该圆与“P1 \-\> P2 连线”的交点，就是他们最终倒下的位置。

*(注：为了简化玩家体验，不需要真的用数学公式算。网页可以提供一个简易的地图工具，或者玩家直接输入两个坐标点后，系统提示“检测到路径，请输入行进距离”)*

### **交互设计 (index.html 底部或 cma.html 的事故上报入口)：**

* **输入框**：\[ 请输入搜救目标推算坐标 \]  
* **错误尝试**：如果玩家直接输入 P1 或 P2，系统提示：“该点已搜索，只有废弃装备。”  
* **正确操作**：  
  * 玩家可能会通过在线地图工具（如 Google Maps 甚至纸笔计算）估算出中间的那个点。  
  * 或者，更简单的设计是：输入框接受**指令式语法**。  
  * 玩家输入：FROM \[34.2140, 108.5500\] TO \[34.2158, 108.5521\] DISTANCE 200m  
* **结局反馈**：  
  * 屏幕闪烁，弹出一张在此坐标拍摄的黑白低像素照片。  
  * 照片内容：一个被冰雪覆盖的深坑（Ice Fissure）。  
  * 文字浮现：“计算匹配。救援队在冰隙下 15 米处发现了他们。全员抱团，保持着体温共享的姿势。未发送的短信显示：‘这里很暖和，我们睡一会儿’。”

**4\. 更加细节的“真实感”打磨**

为了配合你的“细节控”要求，我们加入以下环境叙事：

1. User-Agent 伪装：  
   在 log\_view.html 的请求头（如果玩家会抓包的话）或者页面底部的 debug info 里，显示 User-Agent: NokiaE71/SymbianOS。这暗示了发日志的是一台老式诺基亚手机，解释了为什么只能发纯文本，发不了图。  
2. Base64 的微小瑕疵：  
   解码出来的文本中，可以故意加几个乱码字符（例如 我们只走了 200 就不得不...），模拟数据传输过程中的丢包，增加真实感。  
3. 气象站的侧面印证 (weather.html)：  
   虽然主要解谜不靠它，但气象站数据在 14:45（日志时间）可以显示“风速：0m/s”。这很不正常，暗示他们掉进了一个完全避风的地方——也就是那个冰裂隙深处，侧面印证了你的推理是正确的。
